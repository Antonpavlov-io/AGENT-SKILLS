# Скилл: Pull Requests

## Когда создавать PR

- При любом изменении в `main` — только через PR, не напрямую.
- После завершения работы в feature / fix / hotfix-ветке.

## Структура PR

**Заголовок**: `<prefix>: <краткое описание>` — те же префиксы, что и в коммитах.

**Описание** должно содержать:
- **Что сделано** — список изменений.
- **Зачем** — мотивация или ссылка на задачу.
- **Как проверить** — шаги для ревьюера.
- **Скриншоты** — если затронут UI.

## Правила

1. Один PR — одна задача. Большие изменения разбивать на несколько PR.
2. Перед открытием PR убедиться, что все CI-проверки проходят локально.
3. Назначить ревьюера; без его одобрения не мержить.
4. Стратегия слияния: `Squash and merge` или `Rebase and merge`; `Merge commit` — только если принято в проекте.
5. После слияния ветку удалять.

## Чек-лист перед мержем

- [ ] CI зелёный.
- [ ] Код одобрен ревьюером.
- [ ] Конфликты разрешены.
- [ ] Нет закоммиченных секретов или `.env`-переменных.
- [ ] Документация обновлена (если нужно).
